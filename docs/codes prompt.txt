Small correction before architecture: we probably want Gemma, not Gemini. Gemini is Google’s hosted/API model family; Gemma is the local open-weight family based on Gemini tech. Ollama currently lists gemma4 and gemma3; for an RTX 5080 I’d start with a mid-size quantized Gemma via Ollama and benchmark before going larger.

5 Major Building Blocks

Knowledge Ingestion Pipeline

Converts PDFs, Word files, CSV/JSON/YAML/Excel, Markdown, and existing structured control data into normalized text chunks.

Responsibilities:

Extract text and metadata from documents.
Preserve source references: filename, section, page, framework, control ID, risk category, tier, etc.
Clean OCR/noisy text where needed.
Split content into retrieval-friendly chunks.
Store ingestion status so updates are incremental.
Security Knowledge Model

A structured internal schema for your information-security domain.

This is the layer that turns “random documents” into searchable security knowledge:

Controls/security measures
Risks
Vulnerabilities
Framework mappings: ISO 27001/27002, NIST CSF, NIST 800-53, CIS, etc.
Maturity/tier attributes
Evidence/source provenance
Applicability criteria
I recommend we document this schema early because it becomes the backbone of the RAG quality.

Vector Database + Retrieval Layer

Converts normalized chunks into embeddings and stores them in a local vector database.

Proposed stack:

Python
Ollama embedding model such as nomic-embed-text or mxbai-embed-large
ChromaDB or Qdrant local
Hybrid retrieval if useful: semantic vectors + keyword/filter search
Key requirement: every answer should retrieve from your corpus first, with metadata filters like framework, risk type, tier, business domain, or vulnerability class.

Local LLM Orchestration Layer

The query engine that talks to Ollama/Gemma and forces corpus-grounded answers.

Responsibilities:

Load the local Gemma model through Ollama.
Retrieve relevant chunks before generation.
Build prompts that require citations from your sources.
Refuse or clearly mark answers when the corpus has weak evidence.
Return structured outputs: proposed controls, rationale, source references, confidence, gaps.
Optionally support JSON mode for integration with your app.
Important design rule: the model should behave like a reasoner over your knowledge base, not like a general chatbot.

Project Governance, Tests, and Delivery Workflow

This keeps the implementation durable across restarts, agents, and PRs.

Includes:

AGENT.md as the continuity file with architecture, decisions, commands, repo state, model choices, and next steps.
Unit tests for parsers, chunking, metadata extraction, retrieval, prompt assembly, and response validation.
Integration tests with a tiny sample corpus.
CLI/API entry points for ingestion and querying.
GitHub repository setup.
Regular small PRs: ingestion first, then vector store, then Ollama integration, then app/API layer, then quality evaluation.
My recommended implementation order is:

AGENT.md + repo skeleton
ingestion pipeline
knowledge schema + sample corpus tests
vector DB retrieval
Ollama/Gemma query service
